# Toxic Comment Classification

A multi-label text classification project for the [Jigsaw Toxic Comment Classification Challenge](https://www.kaggle.com/competitions/jigsaw-toxic-comment-classification-challenge) (Kaggle).

Given a comment, the goal is to predict the probability that it falls into each of six
toxicity categories **at the same time**:

`toxic` В· `severe_toxic` В· `obscene` В· `threat` В· `insult` В· `identity_hate`

This is **multi-label**, not binary or multi-class вЂ” a single comment can belong to
several categories simultaneously (e.g. `toxic=1` and `obscene=1` and `insult=1`). Every
model in this repo is built around that constraint: classic baselines use
`OneVsRestClassifier` to train one binary classifier per label, and the neural models use
a sigmoid output layer with binary cross-entropy loss instead of softmax.

## Getting the data

1. Download the dataset from the
   [Kaggle competition page](https://www.kaggle.com/competitions/jigsaw-toxic-comment-classification-challenge/data)
   (free Kaggle account required).
2. Unzip everything. You should end up with `train.csv`, `test.csv`, `test_labels.csv`,
   and `sample_submission.csv` (Kaggle sometimes nests these in their own sub-zips вЂ”
   unzip those too if so).
3. Place all four CSVs directly inside a `data/` folder at the project root (not in
   subfolders).
4. Run `notebooks/00_local_setup.ipynb` to verify everything is found correctly.

## Running this project

Every notebook in `notebooks/` and `models/` auto-detects whether it's
running on **Google Colab** or **locally**, and adjusts data paths, GPU usage, and
batch sizes accordingly вЂ” no manual editing required when switching machines.

**Locally (VSCode / Anaconda):**
```bash
conda create -n toxic-nlp python=3.11
conda activate toxic-nlp
pip install -r requirements.txt
```
Then open any notebook in VSCode and run cells top to bottom. The baselines, EDA, and
data exploration notebooks are CPU-only and run comfortably on a laptop.

**On Google Colab (recommended for BiLSTM and DistilBERT):**
1. Upload the notebook to [colab.research.google.com](https://colab.research.google.com).
2. `Runtime в†’ Change runtime type в†’ T4 GPU в†’ Save`.
3. If using shared data via Drive: in Google Drive, go to **Shared with me**,
   right-click the shared `jigsaw-data` folder в†’ **Add shortcut to Drive**, so it
   becomes reachable at `/content/drive/MyDrive/jigsaw-data/` once mounted.
4. Run cells top to bottom with `Shift+Enter`. The Drive-mount cell will prompt you to
   sign in and authorize access the first time each session.

### Why some models need a GPU and some don't

| | Needs GPU? | Why |
|---|---|---|
| Naive Bayes, Logistic Regression, Linear SVM | No | Linear models on TF-IDF features train in minutes on CPU. |
| BiLSTM | Recommended | Feasible on CPU but considerably slower per epoch. |
| DistilBERT | Practically required | ~66M parameters; full-dataset fine-tuning on CPU can take most of a day, vs. ~30вЂ“60 minutes on a free Colab T4 GPU. |

Baselines, EDA, and data exploration are run locally in this project; BiLSTM and
DistilBERT were trained on Colab's free T4 GPU. Each training notebook saves its
trained weights to Drive (so they survive the Colab session ending), to be copied
locally into `models/`; submission CSVs go into `results/`.

## Evaluation metric

All models are scored with **mean column-wise ROC-AUC** вЂ” the same metric used by the
original Kaggle competition. For each of the 6 labels, ROC-AUC is computed
independently (treating it as its own binary classification problem), and the 6 scores
are averaged into a single number. This is what makes scores from completely different
model types (a linear SVM vs. a transformer) directly comparable.

## Notes on the dataset

- ~159,571 training comments, ~153,164 test comments, no missing values.
- About 9.6% of comments are labeled `toxic`; the other 5 categories are rarer still
  (see `notebooks/02_eda.ipynb` for the full per-label breakdown and label
  co-occurrence analysis).
- `test_labels.csv` marks some rows with `-1`, meaning they were excluded from the
  competition's actual scoring вЂ” these are filtered out before computing evaluation
  metrics on the test set.
